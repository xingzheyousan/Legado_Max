## 1. 灾难性回溯：嵌套量词

### 问题描述

嵌套量词是正则性能的**头号杀手**。当外层量词和内层量词组合时，回溯次数呈指数级增长，导致 CPU 100%、请求超时甚至服务崩溃。

### 危险写法

```regex
(.*)+
(.+)*
(a+)+b
(.*abc)*
```

### 灾难演示

```regex
(a+)+b
```

匹配文本 `aaaaac` 时，引擎会：

1. 第一层 `(a+)` 吃掉所有 `a`
2. 尝试第二层 `(a+)`，失败
3. 回溯：吐出一个 `a`，再试
4. 反复回溯，次数 = 2ⁿ（n 为 `a` 的个数）

📌 这种叫 **灾难性回溯（Catastrophic Backtracking）**

### 优化方案

```regex
# ❌ 嵌套量词
(a+)+b

# ✅ 单一量词 + 明确上限
a{1,100}b

# ✅ 非捕获分组 + 单次量词
(?:abc){0,10}
```

### 核心原则

> **量词里不要再套量词。**

## 2. `.*` 和 `.+` 的滥用

### 问题描述

`.*` 和 `.+` 是最常用的正则语法，也是最容易被滥用的性能陷阱。`.` 匹配一切字符，导致引擎必须逐字符试探，在长文本或多行场景下性能急剧下降。

### 危险写法

```regex
.*error.*
.*@.*
<a>(.*?)</a>
```

### 问题机制

- `.*` 一路贪吃到文本末尾
- 发现后续模式不匹配
- 逐字符回溯，尝试所有可能

### 优化方案

**原则一：缩小匹配范围**

```regex
# ❌ 通配一切
.*@.*

# ✅ 精确字符类
[^@\n]+@[^@\n]+
```

**原则二：明确终止条件**

```regex
# ❌ 无限贪吃
.*error

# ✅ 限定到行尾
.*?error\n
```

**原则三：能不用 `.` 就不用**

```regex
# ❌
.<em>error.</em>

# ✅
[^\n]*error[^\n]*
```

### 核心原则

> **字符类越精确，引擎越快。**

## 3. 大量多选分支 `|` 的效率问题

### 问题描述

`|` 运算符要求引擎按顺序尝试每个分支，分支越多、顺序越不合理，匹配失败时的代价越高。

### 危险写法

```regex
(a|ab|abc|abcd|abcde)x
```

### 问题机制

- 每个分支都要逐一尝试
- 如果文本不匹配，所有分支全部试一遍
- 长文本 × 多分支 = 线性性能灾难

### 优化方案

**方案一：按长度降序排列**

```regex
# ❌ 短到长，多次失败
(a|ab|abc|abcd|abcde)x

# ✅ 长到短，早命中早退出
(abcde|abcd|abc|ab|a)x
```

**方案二：合并公共前缀**

```regex
# ❌ 重复前缀
https?://|https?://www\.

# ✅ 提取公共部分
https?://(?:www\.)?
```

**方案三：用字符类替代简单枚举**

```regex
# ❌
(a|b|c)x

# ✅
[abc]x
```

### 核心原则

> **分支越少越好；不能少就排序；能合并就合并。**

## 4. 捕获分组 `( )` 的过度使用

### 问题描述

每个捕获分组 `( )` 都会让引擎多维护一份回溯状态（保存匹配内容、分配内存）。在循环匹配或处理大文本时，积少成多，成为性能瓶颈。

### 危险写法

```regex
((ab)|(cd)|(ef))(\d+)
```

4 个捕获分组，实际只需要最后的数字。

### 优化方案

**能用非捕获分组就不用捕获分组**

```regex
# ❌ 全部捕获
(ab|cd|ef)(\d+)

# ✅ 仅捕获需要的
(?:ab|cd|ef)(\d+)
```

**只捕获真正需要提取的内容**

```regex
# ❌
(\d{4})-(\d{2})-(\d{2}) (.*)

# ✅ 如果只需要日期
(\d{4}-\d{2}-\d{2}) .*
```

### 核心原则

> **`( )` 捕获 + 记住；`(?: )` 分组 + 不记。不需要记住的，一律 `(?:)`。**

## 5. 在超大字符串上反复匹配

### 问题描述

问题不在正则本身，而在于**调用方式**。在超大字符串上反复调用 `match` / `search`，每次都要重新扫描，时间复杂度叠加。

### 危险写法

```python
# ❌ 逐行反复匹配
for line in huge_text.split('\n'):
    re.search(pattern, line)
```

### 优化方案

**一次匹配，批量提取**

```python
# ✅ 一次 finditer，遍历结果
pattern = re.compile(r'\d+')
for match in pattern.finditer(huge_text):
    process(match)
```

**先切分再匹配**

```python
# ✅ 按块处理，降低单次匹配成本
chunks = split_by_size(huge_text, 10000)
for chunk in chunks:
    pattern.finditer(chunk)
```

### 核心原则

> **正则不怕复杂，怕的是反复启动。编译一次，复用到底。**

## 6. 锚点缺失导致无谓扫描

### 问题描述

缺少锚点（`^`、`$`、`\b`）时，引擎必须从字符串第一个字符开始逐位尝试，即使目标内容只可能出现在特定位置。

### 危险写法

```regex
error
```

在 1MB 日志中：
- 从第 1 个字符开始试
- 失败 → 移 1 位 → 再试
- 重复 N 次

### 优化方案

**使用行首锚点**

```regex
# ✅ 只在行首匹配
^error
```

**使用行尾锚点**

```regex
# ✅ 只在行尾匹配
error$
```

**使用单词边界**

```regex
# ✅ 精确匹配整个单词
\berror\b
```

**使用环视断言**

```regex
# ✅ 匹配后面紧跟冒号的 error
error(?=:)
```

### 核心原则

> **锚点 = 直接告诉引擎别瞎找。能用锚点就用锚点。**

## 7. `.*?` 非贪婪匹配的误用

### 问题描述

很多人以为 `.*?` 是万能药，不会造成性能问题。事实上，非贪婪只是**换了个方向试探**，仍然要逐字符回溯，多层嵌套时照样慢。

### 危险写法

```regex
<a>(.*?)</a>
<div>(.*?)</div>
```

### 问题机制

- `.*?` 逐字符扩展，每扩展一次就检查一次后续模式
- 嵌套层级越多，试探次数越多
- HTML/XML 场景下尤为严重

### 优化方案

**用字符类拒绝不需要的字符**

```regex
# ❌ 非贪婪但仍在试探
<a>(.*?)</a>

# ✅ 直接拒绝 '<'，一步到位
<a>([^<]*)</a>
```

**用更精确的模式替代通配**

```regex
# ❌
<div>(.*?)</div>

# ✅
<div>([^<]*)</div>
```

### 核心原则

> **能拒绝，就不要慢慢试。**

## 8. Unicode / 中文场景的隐形成本

### 问题描述

Unicode 属性判断（如 `\w`、`\s`、`.`）比 ASCII 字符类慢得多，尤其在中文混合换行、全角半角混排的场景下。

### 危险写法

```regex
[\s\S]*       # 匹配一切（含 Unicode 判定）
.              # 默认包含 Unicode
```

### 优化方案

**用明确的字符范围替代通用类**

```regex
# ❌ 隐式 Unicode 判定
[\s\S]*

# ✅ 明确排除换行
[^\n]*
```

**用 Unicode 码点替代模糊匹配**

```regex
# ❌
.

# ✅ 只匹配中文字符
[\u4e00-\u9fa5]
```

**统一全角/半角处理**

```python
# 预处理：全角转半角，降低正则复杂度
text = text.replace('（', '(').replace('）', ')')
```

### 核心原则

> **中文场景下，越明确的字符集 = 越快的正则。**

## 9. 在循环中重复编译正则

### 问题描述

每次调用正则匹配时都重新编译模式，编译开销远大于匹配开销，在高频循环中是致命的。

### 危险写法

```python
# ❌ 每次循环都编译
for i in data:
    re.match(r"\d+", i)
```

### 优化方案

**编译一次，重复使用**

```python
# ✅ 提前编译
pattern = re.compile(r"\d+")
for i in data:
    pattern.match(i)
```

**封装为常量**

```python
# ✅ 模块级常量，程序生命周期内只编译一次
PHONE_PATTERN = re.compile(r"1[3-9]\d{9}")
EMAIL_PATTERN = re.compile(r"[^@\n]+@[^@\n]+")
```

### 核心原则

> **正则编译是重量级操作，匹配是轻量级操作。别把重操作放循环里。**

## 10. 缺少回溯控制导致指数级匹配

### 问题描述

某些正则写法虽然表面上没有嵌套量词，但由于缺少回溯控制，引擎仍然会尝试大量无效路径。

### 危险写法

```regex
# 匹配 URL，但没有限制协议头的次数
(https?://)+.*$
```

### 优化方案

**使用原子分组（Atomic Group）**

```regex
# ✅ 一旦匹配成功，不允许回溯
(?>https?://)[^ ]*
```

**使用占有量词（Possessive Quantifier）**

```regex
# ✅ 不回溯，一步到位
.*+         # Java 支持
[^\n]++     # Java 支持
```

**手动切断回溯路径**

```regex
# ❌ 可能反复回溯
(a+)+b

# ✅ 明确匹配上限
a{1,100}b
```

### 语言支持对照

| 特性 | Python | JavaScript | Java |
|---|---|---|---|
| 原子分组 `(?>)` | ❌ | ❌ | ✅ |
| 占有量词 `++` | ❌ | ❌ | ✅ |
| 非贪婪 `?` | ✅ | ✅ | ✅ |

### 核心原则

> **能切断回溯路径的，就不要留给引擎去试探。**

## 11. 正则引擎选型不当

### 问题描述

不同语言的正则引擎性能差异巨大。用错引擎或用错模式，可能差几个数量级。

### 各语言正则引擎特点

| 语言 | 引擎类型 | 特点 |
|---|---|---|
| **Python `re`** | 回溯引擎 | 功能全，但复杂正则慢 |
| **Python `regex`** | 回溯引擎（增强） | 支持原子分组、占有量词 |
| **JavaScript** | 回溯引擎 | 不支持原子分组/占有量词 |
| **Java `Pattern`** | 回溯引擎 | 支持原子分组、占有量词 |
| **Go `regexp`** | RE2（DFA） | 无回溯，但功能受限 |
| **Rust `regex`** | 类 DFA | 无回溯，性能极佳 |

### 优化建议

**场景一：需要极致性能 + 不需要高级特性**

```go
// ✅ Go 的 regexp（基于 RE2），无回溯风险
re := regexp.MustCompile(`\d+`)
```

**场景二：需要高级特性 + 注意性能**

```python
# Python re 模块够用，但复杂场景考虑 regex 第三方库
import regex
pattern = regex.compile(r'(?>a+)b')
```

**场景三：前端 JavaScript**

```javascript
// JS 不支持原子分组，用字符类替代
// ❌ 容易回溯
const bad = /(a+)+b/;

// ✅ 限定范围
const good = /a{1,100}b/;
```

### 核心原则

> **简单匹配用 DFA 引擎（Go / Rust），复杂匹配用回溯引擎但要控制回溯深度。**

## 速查表：红榜 vs 黑榜

| 类别 | 黑榜 ❌ | 红榜 ✅ |
|---|---|---|
| 量词 | `(.*)*` | `.{0,100}` |
| 通配 | `.` | `[^\n]` |
| 分组 | `(a\|b\|c)` | `(?:a\|b\|c)` |
| 多选 | 长分支乱序 | 合并 / 排序 |
| 匹配 | 循环 match | 一次 finditer |
| 锚点 | 无 | `^` `$` `\b` |
| HTML | `.*?</tag>` | `[^<]*` |
| 编译 | 循环内编译 | 提前编译复用 |
| Unicode | `[\s\S]` | `[^\n]` |
| 回溯控制 | 无 | 原子分组 / 上限 |

## 一句话总结

> **正则慢，99% 不是引擎慢，而是：给的选择太多、限制太少、试错成本太高。**
>
> **写正则的三问：**
> 1. 能限定范围吗？
> 2. 能设上限吗？
> 3. 能少一个分支吗？
